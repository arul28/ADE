## What is ADE?

ADE (Agentic Development Environment) is a **macOS desktop application** that serves as the orchestration layer between you and your AI coding agents. While agents write code across multiple git branches simultaneously, ADE watches over them — tracking context, predicting conflicts, surfacing integration risks, managing budgets, and keeping everything synchronized.

Think of it as a cockpit for agentic development. You set direction; ADE coordinates execution.

## The Problem ADE Solves

Modern AI coding agents are fast — but running several at once against the same repository quickly turns into chaos:

- **Conflicting changes**: Two agents edit the same files in different branches. By the time you notice, the conflict is deep.
- **Lost context**: An agent starting a new task has no memory of what other agents did this week. Every session starts cold.
- **No oversight layer**: There is no single place to see what all your agents are doing, what they spent, or whether they succeeded.
- **Merge risk is invisible**: You cannot see that two branches will conflict until you actually try to merge them.

ADE addresses every one of these problems with a purpose-built architecture: isolated workspaces (Lanes), durable context bundles (Packs), continuous merge simulation (Conflict Engine), an always-on project agent (the CTO), and a full audit trail (History, Missions).

## How ADE Is Different

<AccordionGroup>
  <Accordion title="Not a code editor" icon="pen-nib">
    ADE is not trying to replace VS Code, Cursor, or Neovim. It has a built-in file viewer and editor for quick inspections, but its job is **orchestration**, not text editing. You keep using your preferred editor; ADE manages the agents running alongside it.
  </Accordion>
  <Accordion title="Not a terminal multiplexer" icon="rectangle-terminal">
    ADE wraps terminal sessions with rich metadata — agent attribution, session transcripts, timing data, and delta computation — so every shell command is traceable. It is not a replacement for tmux, but it makes your terminals far more observable.
  </Accordion>
  <Accordion title="Not a CI/CD system" icon="arrow-rotate-left">
    ADE runs locally on your machine. It has no remote build agents, no YAML pipelines, and no cloud control plane. It is designed to be the **local counterpart** to your CI system — the place where AI work happens before it reaches CI.
  </Accordion>
  <Accordion title="Not a chat interface" icon="comment">
    ADE includes an Agent Chat pane for direct conversation with AI models inside a lane, but the Chat view is one of twelve integrated views, not the center of the product. ADE's strength is in the orchestration layer above the chat: Missions, the CTO, Automations, and the Conflict Engine.
  </Accordion>
</AccordionGroup>

## Architecture: Local-First, Privacy-First

ADE is built on a strict **trust boundary model** that keeps your API keys, source code, and agent outputs on your machine.

```
┌─────────────────────────────────────────────────────────┐
│  Renderer Process (Untrusted UI)                        │
│  React 18 · TypeScript · Tailwind CSS                   │
│  ┌──────────────────────────────────────────────────┐   │
│  │  Preload Bridge (contextBridge)                  │   │
│  │  Typed IPC channel — no Node.js access           │   │
│  └──────────────────────────────────────────────────┘   │
├─────────────────────────────────────────────────────────┤
│  Main Process (Trusted — full Node.js access)           │
│  Electron 40.x · SQLite · All services                  │
│  API keys · File system · Git operations · PTY          │
└─────────────────────────────────────────────────────────┘
```

<CardGroup cols={3}>
  <Card title="Renderer: Untrusted UI" icon="display">
    The React UI cannot access the file system, spawn processes, or read secrets. It communicates only through the typed IPC bridge.
  </Card>
  <Card title="Preload Bridge" icon="arrow-left-right">
    The `contextBridge` layer exposes a strict API surface. Every call is validated and routed through registered IPC handlers.
  </Card>
  <Card title="Main Process: Trusted Core" icon="shield-check">
    All services run here: git, PTY sessions, AI provider calls, SQLite, MCP server. API keys never leave this boundary.
  </Card>
</CardGroup>

<Note>
All project data is stored in `.ade/` at the root of your repository and in a local SQLite database on your machine. Nothing is sent to ADE servers. AI API calls go directly from the main process to your chosen provider (Anthropic, OpenAI, etc.).
</Note>

## The MCP Server

ADE includes a built-in **MCP (Model Context Protocol) server** that exposes its capabilities to AI agents as callable tools via JSON-RPC 2.0. When an agent is running inside a lane, it can call ADE tools to:

- Spawn other agents in specific lanes
- Read and write files across worktrees
- Execute git operations (commit, push, rebase, merge-simulate)
- Create and update missions and their steps
- Query session transcripts and deltas
- Manage context packs
- Route computer-use artifacts to missions, PRs, or Linear issues
- Ask the user a structured question (with rate limiting)

The MCP server runs in the main process and authenticates each session by role (`orchestrator`, `agent`, `external`, `evaluator`). Tools are scoped accordingly — coordinator-level tools are only visible to orchestrator-role callers.

<Tip>
The MCP server endpoint is available to any process that knows the socket path. This means you can connect Cursor, Claude Desktop, or any other MCP-compatible client to ADE and drive it programmatically.
</Tip>

## How It All Fits Together

Here is a typical flow from a new task to a merged PR:

<Steps>
  <Step title="Open a Lane">
    You (or the CTO agent) create a new Worktree Lane. ADE runs `git worktree add` under `.ade/worktrees/` and sets up the lane's process environment.
  </Step>
  <Step title="Load Context">
    The lane's Pack is injected into the agent's system prompt. The Pack contains architecture notes, coding conventions, risk signals, and the lane's stated intent.
  </Step>
  <Step title="Agent Runs">
    You start an agent chat or a Mission. The agent operates inside the lane's worktree, calling ADE's MCP tools for git operations, file edits, and session management.
  </Step>
  <Step title="Conflict Engine Monitors">
    In the background, the Job Engine periodically runs merge simulations between all active lanes. If two lanes touch the same files, a Conflict record is created and surfaced in the Conflicts view.
  </Step>
  <Step title="Mission Validates">
    When execution completes, the Mission's Validation phase runs — a dedicated validator agent reviews the changes, checks against the original goal, and may collect computer-use proof artifacts (screenshots, browser traces).
  </Step>
  <Step title="PR Is Created">
    The agent (or ADE automatically via an Automation) opens a GitHub PR from the lane's branch. ADE links the PR back to the Mission, Linear issue, and any collected artifacts.
  </Step>
  <Step title="CTO Synthesizes">
    The CTO agent sees the completed mission, updates its project memory, adjusts budgets, and either queues the next task or surfaces a summary to you.
  </Step>
</Steps>
