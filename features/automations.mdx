## Overview

Automations are ADE's background execution layer. Each automation is a simple rule:

> **Trigger** → **Executor**

When a trigger fires (a PR is opened, a commit lands, a Linear issue is created, or a cron schedule fires), ADE hands control to an AI executor that runs with full tool access. The executor works in the background — you do not need to watch it, approve each step, or even be at your machine.

Think of Automations as GitHub Actions, but instead of a shell script, the "action" is a fully capable AI agent that can read your codebase, write files, commit, open PRs, and update Linear issues.

## Trigger Types

### Local Git & File Events

| Trigger | Description | Filter Options |
|---|---|---|
| `git.push` | Code is pushed to any branch | Branch name or glob pattern |
| `git.pr_opened` | A PR is opened from any lane | Source branch pattern |
| `git.pr_merged` | A PR is merged | Target branch |
| `git.pr_closed` | A PR is closed without merging | Source branch pattern |
| `git.commit` | A commit is made locally | File path filter (glob) |
| `file.change` | A specific file or glob changes on disk | File path glob |
| `lane.created` | A new lane is created | Lane name pattern |
| `lane.archived` | A lane is archived | Lane name pattern |
| `schedule` | Cron expression | Any valid cron expression |
| `manual` | User clicks "Run" in the Automations UI | — |

### External Events (Linear)

| Trigger | Description | Filter Options |
|---|---|---|
| `linear.issue_created` | A new Linear issue is created | Team, label, project |
| `linear.issue_updated` | An issue is updated | Status transition, field changed |
| `linear.issue_assigned` | An issue is assigned | Assignee (person or ADE bot) |

<Note>
Linear triggers require an active Linear integration. See [Configuration → Linear](/configuration/settings) to connect your workspace. ADE distinguishes between reading Linear (always allowed) and writing Linear — the `linear.write` permission must be explicitly enabled per automation.
</Note>

## Executor Modes

Each automation uses one of four executor modes. Choosing the right mode affects cost, persistence, and scheduling.

<AccordionGroup>
  <Accordion title="automation-bot — Lightweight Stateless Executor" icon="robot">
    A fast, lightweight executor designed for simple, contained tasks. Each run starts fresh with no memory of previous executions. Ideal for tasks that are self-contained and predictable.

    **Best for:** Running tests on every commit, summarizing new commits to a Slack message, formatting changed files, generating changelog entries.

    **Characteristics:**
    - Stateless — each run is independent
    - Low latency startup
    - Best cost efficiency for repetitive tasks
  </Accordion>

  <Accordion title="employee — Persistent Worker Agent" icon="user-tie">
    A persistent agent assigned to own a specific type of task. The employee has a memory store and remembers what it has seen across multiple runs. Use this when you need continuity — the agent should accumulate knowledge over time.

    **Best for:** Reviewing all CSS changes (the agent learns your style preferences over time), maintaining the test suite, tracking dependency debt.

    **Characteristics:**
    - Stateful memory across runs
    - Builds domain knowledge over time
    - Higher initial cost, but more accurate over time
  </Accordion>

  <Accordion title="cto-route — CTO Agent Decision Routing" icon="arrow-up-right-from-square">
    Routes the trigger event to the CTO agent for high-level decision making. The CTO agent processes the event using its full project context and persistent memory, then decides what to do: spawn a mission, update a plan, comment on a PR, or do nothing.

    **Best for:** Linear issue triage (CTO decides priority and assignment), milestone risk assessment, cross-lane coordination decisions.

    **Characteristics:**
    - Leverages the CTO's full project memory
    - Produces high-context, high-quality decisions
    - Higher cost per execution
  </Accordion>

  <Accordion title="night-shift — Off-Hours Deferred Execution" icon="moon">
    Queues the automation to run during off-hours when your machine is idle. ADE batches Night Shift tasks and processes them overnight. This mode is ideal for expensive, non-urgent tasks that do not need immediate results.

    **Best for:** Weekly dependency updates, large codebase refactoring sweeps, generating comprehensive documentation, model evaluation runs.

    **Characteristics:**
    - Deferred to idle time (default: 15 minutes of inactivity)
    - Tasks processed one at a time to avoid resource contention
    - Results available when you return the next morning
  </Accordion>
</AccordionGroup>

## Creating Your First Automation

<Info>**📸 Screenshot needed:** The "New Automation" dialog showing the trigger type selector, executor mode radio buttons, the automation prompt text area, and the tool palette dropdown. The form should look partially filled-in.</Info>

<Steps>
  <Step title="Open the Automations Tab">
    Click the **Automations** icon in the left sidebar (the lightning bolt icon). Then click **New Automation** in the top-right corner.
  </Step>
  <Step title="Choose a Trigger">
    Select a trigger family from the dropdown, then configure its filters. For example, for `git.pr_opened`, you can filter to only fire when the source branch matches `feat/*`.
  </Step>
  <Step title="Select an Executor Mode">
    Choose between `automation-bot`, `employee`, `cto-route`, or `night-shift` based on the nature of the task. When in doubt, start with `automation-bot`.
  </Step>
  <Step title="Write the Automation Prompt">
    This is the instruction the executor receives when the trigger fires. Be specific about what the agent should produce. Example:

    ```
    A PR was just opened. Review the diff for:
    1. Any hardcoded credentials or secrets
    2. Missing error handling in async functions
    3. Files that changed but have no corresponding test changes

    Post a PR comment summarizing your findings. If you find secrets, mark the PR as "Needs Changes".
    ```
  </Step>
  <Step title="Configure Tool Palette">
    Choose which tools the executor can access. For a PR reviewer, `read-only` is sufficient — the agent needs to read files but should not write anything. See [Tool Palette Security](#tool-palette-security) below.
  </Step>
  <Step title="Set Guardrails & Budget">
    Set a maximum token budget and execution time limit. Enable **Approval Mode** if you want the automation to pause before any write operations.
  </Step>
  <Step title="Save and Enable">
    Click **Save**. The automation is now active. Toggle the enable switch to pause it at any time without deleting it.
  </Step>
</Steps>

## Tool Palette Security

Every automation runs with a restricted set of tools called a **palette**. This is a security best practice — an automation that runs unattended should only have access to exactly what it needs.

| Palette | Capabilities | When to Use |
|---|---|---|
| `read-only` | Read files, read git history, read context packs | PR review, summarization, analysis |
| `write-files` | Everything in read-only + write and create files | Auto-formatting, documentation generation |
| `full-git` | Everything in write-files + all git operations | Auto-committing, branch management |
| `full-ade` | All ADE tools including Linear and mission management | Complex orchestration automations |
| Custom | Explicitly listed tools | Fine-grained control |

<Warning>
Avoid granting `full-ade` permissions to automations that run on external triggers like `linear.issue_created`. A malformed or adversarial issue body could attempt prompt injection. Use the minimum palette required for the task.
</Warning>

## Memory Modes

Control how much state an automation carries between runs.

<Tabs>
  <Tab title="stateless">
    Each execution starts completely fresh. The automation has no knowledge of previous runs. This is the default and the safest option.

    ```yaml
    memory: stateless
    ```
  </Tab>
  <Tab title="stateful">
    The automation has a persistent memory store scoped to that automation. Useful for tracking state across executions — for example, "remember the last PR you reviewed so you don't repeat yourself."

    ```yaml
    memory: stateful
    ```
  </Tab>
  <Tab title="project">
    The automation reads from and writes to the project-level memory shared with the CTO and other automations. Use this when your automation needs to coordinate with other agents or influence CTO decisions.

    ```yaml
    memory: project
    ```
  </Tab>
</Tabs>

## Night Shift Queue

<Info>**📸 Screenshot needed:** The Automations → Night Shift tab showing a queue of pending tasks with their estimated cost, priority order, and a "Run Night Shift Now" button. One task should show as "Completed at 2:14 AM" with a green checkmark.</Info>

The Night Shift system allows you to defer expensive, non-urgent automations to run while your machine is idle.

**How it works:**

1. Any automation with executor mode `night-shift` adds to the queue when its trigger fires.
2. ADE monitors system idle time. After **15 minutes** of no keyboard or mouse input, Night Shift processing begins.
3. Tasks are processed **one at a time** to avoid CPU/memory contention with background processes.
4. Results (commits, PRs, comments, notifications) are available when you return.

**Managing the Queue:**

- Navigate to **Automations → Night Shift** to see the current queue, estimated costs, and history of past runs.
- Click **Run Night Shift Now** to process the queue immediately, regardless of idle state.
- Individual tasks can be removed from the queue or reprioritized by dragging.

<Tip>
Schedule resource-intensive tasks like dependency updates or full test suite runs as Night Shift automations. They will complete overnight without competing with your daytime work.
</Tip>

## Automation Templates

ADE ships with the following built-in templates. Enable them from **Automations → Templates**.

<CardGroup cols={2}>
  <Card title="PR Reviewer" icon="code-pull-request">
    **Trigger:** `git.pr_opened`
    **Executor:** `automation-bot`

    Automatically reviews every PR opened in ADE. Checks for missing error handling, untested code paths, and obvious logic bugs. Posts a structured review comment.
  </Card>
  <Card title="Test Guardian" icon="shield">
    **Trigger:** `git.commit`
    **Executor:** `automation-bot`

    Runs your test suite on every commit. If tests fail, posts a comment on the associated PR with the failure summary and a suggested fix.
  </Card>
  <Card title="Linear Triager" icon="list-check">
    **Trigger:** `linear.issue_created`
    **Executor:** `cto-route`

    Routes new Linear issues to the CTO agent for triage. The CTO assigns priority, estimates complexity, and optionally creates a new lane for the work.
  </Card>
  <Card title="Daily Standup" icon="sun">
    **Trigger:** `schedule` — `0 9 * * 1-5`
    **Executor:** `automation-bot`

    Every weekday morning, generates a summary of all lane activity from the previous day: commits made, PRs opened/merged, missions completed, and any active conflicts.
  </Card>
  <Card title="Dependency Updater" icon="arrow-rotate-right">
    **Trigger:** `schedule` — `0 2 * * 0` (weekly, Sunday 2am)
    **Executor:** `night-shift`

    Weekly check for outdated dependencies. Opens a draft PR with the proposed updates and a compatibility assessment for each changed package.
  </Card>
</CardGroup>

## Outputs and Routing

Automations can produce the following output types. Configure output routing in the automation settings to control where results land.

| Output Type | Description |
|---|---|
| Commits | Committed changes to a branch in a specific lane |
| Pull Requests | New or updated PRs on GitHub |
| Linear Updates | Issue status changes, comments, assignments |
| Notifications | In-ADE toast notifications or macOS system notifications |
| Lane Pack Updates | Triggers a refresh of a lane's context pack |
| Chat Messages | Posts a message into a specific lane's chat |

## Guardrails Reference

| Guardrail | Description |
|---|---|
| Budget Cap | Maximum token cost per execution. Automation halts if exceeded. |
| Time Limit | Maximum wall-clock execution time. Halts and reports timeout. |
| File Path Restriction | Automation can only read/write files matching a specified glob pattern. |
| Dry-Run Mode | Shows what the automation would do without performing any writes. |
| Approval Mode | Automation pauses and requests your approval before any write operation. |

## Troubleshooting

<AccordionGroup>
  <Accordion title="Automation is not firing" icon="circle-question">
    1. Confirm the automation is **enabled** — check the toggle in the Automations list.
    2. Verify the trigger filter is not too restrictive. For example, a `git.push` trigger with `branch: release/*` will not fire on pushes to `main`.
    3. For Linear triggers, confirm the Linear integration is connected and the webhook is registered: **Settings → Integrations → Linear → Webhook Status**.
    4. Check the automation's **Run History** tab for any error entries — trigger evaluation errors are logged there even if the executor never ran.
  </Accordion>

  <Accordion title="Budget exceeded — automation halted" icon="circle-dollar-to-slot">
    The automation hit its configured token budget cap and stopped mid-execution. To resolve:

    1. Review the partial results in **Run History** to understand how far it got.
    2. Increase the budget cap in the automation's settings if the task genuinely requires more tokens.
    3. Alternatively, break the automation's prompt into a smaller, more focused task.
    4. For recurring budget overruns, switch to `night-shift` mode which has a separate, higher default budget ceiling.
  </Accordion>

  <Accordion title="Guardrail blocked a file write" icon="lock">
    The automation attempted to write a file outside its configured file path restriction. This is the guardrail working as intended.

    If the write was legitimate, update the **File Path Restriction** glob in the automation settings to include the path that was blocked. If unexpected, investigate the automation prompt — the agent may be hallucinating file paths.
  </Accordion>

  <Accordion title="Night Shift queue is not processing" icon="moon">
    Night Shift requires your machine to be idle for 15 minutes (configurable in **Settings → Automations → Idle Threshold**). Common reasons the queue does not process:

    - A background process (e.g., video call, build process) is keeping the machine active.
    - The ADE app was closed before idle threshold was reached.
    - Manual "Run Night Shift Now" button is always available as an override.
  </Accordion>

  <Accordion title="Linear automation writing without permission" icon="triangle-exclamation">
    By default, automations can only **read** Linear data. To allow writes (comments, status changes, assignments), you must explicitly enable the `linear.write` permission in the automation's settings under **Permissions → External Services**.
  </Accordion>
</AccordionGroup>
