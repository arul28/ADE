# W9: OpenClaw Bridge (External Agent Gateway)
> Source: OpenClaw integration architecture from [features/CTO.md § OpenClaw Integration Architecture](../../features/CTO.md). [OpenClaw Memory Masterclass](https://velvetshark.com/articles/openclaw-memory-masterclass) — sub-agent memory inheritance, bootstrap file reload. [OpenClaw Gateway Protocol](https://docs.openclaw.ai/concepts/gateway) — device pairing, WebSocket operator protocol, session routing.

Dependencies: **W1** (CTO agent core), **W8** (external MCP — bridge uses MCP transport patterns).

Connect ADE's CTO agent to OpenClaw (or similar external agent gateways), enabling bidirectional communication between the user's personal agent network and ADE's tech-focused CTO.

##### Conceptual Model

- OpenClaw is the user's personal life gateway — multiple agents handling different domains (finance, health, scheduling, etc.).
- ADE CTO is the entire tech department — one persistent agent with deep project knowledge.
- The bridge makes CTO accessible as a specialist within the user's OpenClaw agent network.
- Communication is bidirectional: OpenClaw agents can ask CTO for tech advice/status, and CTO can proactively notify OpenClaw agents about project events (deploy complete, CI broken, mission finished).

##### Bridge Service

`openclawBridgeService.ts` in `src/main/services/`:

- **HTTP server** (port 3742): Receives inbound requests from OpenClaw hooks/skills. Acknowledges immediately with 202, forwards to CTO via IPC, returns response when CTO completes processing. Timeout: 120s for CTO processing, return 504 if exceeded.
- **WebSocket operator client**: Connects to OpenClaw's Gateway (`ws://127.0.0.1:18789`) as `operator` role for bidirectional, low-latency communication.
- Handles OpenClaw's device pairing protocol: challenge-nonce handshake, `connect` request with auth token, `deviceToken` persistence for reconnection.
- Subscribes to `agent` and `chat` events for real-time message routing.
- **Idempotency**: Each inbound message carries an `idempotencyKey`. Bridge stores processed keys in a 24-hour TTL cache. Duplicate messages return the cached response without re-invoking CTO.

##### Memory Bridge

Adapted from [OpenClaw's sub-agent memory inheritance](https://velvetshark.com/articles/openclaw-memory-masterclass): CTO can share relevant project context with OpenClaw agents, and receive context from them.

- **Outbound context sharing**: When CTO responds to an OpenClaw query, it can attach relevant project memory entries as structured context. The bridge serializes these as OpenClaw-compatible `context` blocks in the response payload.
- **Inbound context receiving**: When OpenClaw agents send messages to CTO, they can include context blocks (e.g., a scheduling agent telling CTO about an upcoming deadline). The bridge converts these to ephemeral mission memory entries (`scope: "mission"`, auto-archived after 24h).
- **Selective sharing**: CTO identity config includes `openclawContextPolicy`:
  ```typescript
  openclawContextPolicy: {
    shareProjectFacts: boolean;    // share non-sensitive project facts
    shareCodePatterns: boolean;    // share coding conventions
    neverShare: string[];          // categories to never share: ["secret", "credential"]
  }
  ```

##### Communication Flows

- **Inbound (OpenClaw → CTO)**: OpenClaw agent calls `sessions_send` targeting `hook:ade-cto` → Gateway hook POSTs to bridge → bridge checks idempotency → forwards to CTO via IPC → CTO processes (with OpenClaw context injected) → bridge POSTs reply back with optional project context.

- **Outbound (CTO → OpenClaw)**: CTO/orchestrator wants to proactively message OpenClaw agent → bridge calls `sessions_send` over WebSocket → target agent receives and replies stream back as `agent` events. Use cases:
  - Mission completed → notify scheduling agent
  - CI broken → notify relevant agents
  - Deploy successful → notify status agents

- **Fallback: skill-only bridge** (no WebSocket): Simpler one-directional integration where OpenClaw agents use `exec` + `curl` to hit ADE's HTTP endpoint. No proactive notifications in this mode.

##### IPC Channels

| Channel | Direction | Purpose |
|---------|-----------|---------|
| `openclaw:message-received` | main → renderer | Inbound message from OpenClaw to CTO |
| `cto:forward-to-openclaw` | renderer → main | Outbound message from CTO to OpenClaw agent |
| `openclaw:connection-status` | main → renderer | Bridge connection state for UI indicator |
| `openclaw:message-history` | renderer → main | Fetch recent OpenClaw ↔ CTO messages |

##### Configuration

In `.ade/local.secret.yaml` (gitignored):
```yaml
openclaw:
  enabled: false
  gatewayUrl: ws://127.0.0.1:18789
  gatewayToken: ${OPENCLAW_GATEWAY_TOKEN}
  bridgePort: 3742
  hooksToken: <shared-secret>
  deviceToken: <persisted-after-first-handshake>
  contextPolicy:
    shareProjectFacts: true
    shareCodePatterns: true
    neverShare: ["secret", "credential"]
```

##### Scope Boundary

ADE implements the bridge service. OpenClaw-side configuration (hooks, skills, agent setup) is user-managed and documented in `docs/features/CTO.md`.

##### Renderer

- **CTO tab > OpenClaw panel**: Connection status indicator (connected/disconnected/reconnecting), recent message history, "Send to OpenClaw" action, context sharing policy toggle.
- **Settings > Integrations > OpenClaw**: Bridge configuration, connection test button, device pairing status, context policy editor.
- **Activity feed**: OpenClaw interactions appear in CTO activity feed with an OpenClaw badge, showing inbound queries and outbound notifications.

**Tests:**
- HTTP endpoint: inbound message acceptance (202 acknowledged), async CTO forwarding, response delivery within timeout.
- Idempotency: duplicate inbound message returns cached response without re-invoking CTO.
- WebSocket operator: handshake (challenge-nonce, connect, hello-ok), reconnection with persisted `deviceToken`, graceful reconnection on connection drop.
- Outbound: `sessions_send` over WebSocket delivers to OpenClaw and receives reply events.
- Memory bridge: outbound response includes project context entries, inbound context converted to ephemeral mission memory, `neverShare` categories filtered from outbound context.
- Configuration: `.ade/local.secret.yaml` openclaw section parsing, enabled/disabled toggle, secret isolation, context policy applied.
- Error handling: OpenClaw gateway unavailable (graceful degradation, UI shows disconnected), CTO unavailable (queued retry with 3 attempts), malformed inbound requests (reject with 400), timeout on CTO processing (504 returned).
- Fallback mode: skill-only bridge works without WebSocket, inbound HTTP works, outbound notifications disabled.
