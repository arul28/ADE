# W9: OpenClaw Bridge (External Agent Gateway)
> Source: OpenClaw integration architecture from [features/CTO.md § OpenClaw Integration Architecture](../../features/CTO.md). [OpenClaw Memory Masterclass](https://velvetshark.com/articles/openclaw-memory-masterclass) — sub-agent memory inheritance, bootstrap file reload. [OpenClaw Gateway Protocol](https://docs.openclaw.ai/concepts/gateway) — device pairing, WebSocket operator protocol, session routing.

Dependencies: **W1** (CTO agent core), **W8** (implemented ADE-managed external MCP substrate).

W9 should treat W8 as a shipped platform boundary, not a planned one. External third-party tool access is already centralized in ADE and re-exposed through ADE MCP as `ext.<server>.<tool>`. W9 should build on that ADE-owned surface rather than inventing a second provider-specific external-MCP path.

Connect ADE's CTO agent to OpenClaw (or similar external agent gateways), enabling bidirectional communication between the user's personal agent network and ADE's tech-focused CTO.

##### Conceptual Model

- OpenClaw is the user's personal life gateway — multiple agents handling different domains (finance, health, scheduling, etc.).
- ADE CTO is the entire tech department — one persistent agent with deep project knowledge.
- The bridge makes CTO accessible as a specialist within the user's OpenClaw agent network.
- Communication is bidirectional: OpenClaw agents can ask CTO for tech advice/status, and CTO can proactively notify OpenClaw agents about project events (deploy complete, CI broken, mission finished).

##### Bridge Service

`openclawBridgeService.ts` in `apps/desktop/src/main/services/cto/`:

- **HTTP server** (`/openclaw/health`, `/openclaw/hook`, `/openclaw/query`): Runs on loopback only. Default port is `18791`, and it is configurable from `.ade/local.secret.yaml`.
- **Hook mode**: `POST /openclaw/hook` validates the shared hook token, marks idempotency, returns `202` immediately, then forwards the request to CTO asynchronously and delivers the reply back through the OpenClaw Gateway WebSocket operator session.
- **Query mode**: `POST /openclaw/query` provides the simple skill and `curl` fallback. It waits for the ADE reply and returns `200`, or `504` if the bridge turn times out.
- **WebSocket operator client**: Connects to OpenClaw Gateway (`ws://127.0.0.1:18789` by default) as an `operator` client for pairing, proactive outbound sends, queued reply delivery, and reconnect handling.
- Handles OpenClaw's actual pairing handshake: `connect.challenge` event, signed `connect` request, persisted `deviceToken`, bounded reconnect backoff, and queued outbox replay after reconnect.
- **Idempotency**: Each inbound request uses `requestId` or `idempotencyKey`. The bridge stores processed keys in a 24-hour TTL cache under `.ade/cache/openclaw/openclaw-idempotency.json`.
- **Persistence**: Small bridge state is split by ownership:
  - `.ade/cto/openclaw-device.json` stores the paired device identity because it is durable CTO identity state.
  - `.ade/cache/openclaw/` stores runtime bridge files and migrates legacy repo-visible copies on startup:
    - `openclaw-history.json`
    - `openclaw-outbox.json`
    - `openclaw-idempotency.json`
    - `openclaw-routes.json`

##### Context Sharing

Adapted from [OpenClaw's sub-agent memory inheritance](https://velvetshark.com/articles/openclaw-memory-masterclass): CTO can share relevant project context with OpenClaw agents, and receive context from them without automatically polluting durable ADE memory.

- **Outbound context sharing**: When ADE sends a reply or a proactive notification back into OpenClaw, the bridge can attach structured context from the current turn. The bridge filters that payload through CTO identity `openclawContextPolicy` before sending.
- **Inbound context receiving**: When OpenClaw agents send context blocks, the bridge injects them into the CTO prompt as turn-scoped bridge metadata. They are recorded in bridge history but are not auto-promoted into durable CTO memory on receipt.
- **Selective sharing**: CTO identity config includes `openclawContextPolicy`:
  ```typescript
  openclawContextPolicy: {
    shareMode: "full" | "filtered";
    blockedCategories: string[];
  }
  ```

##### Communication Flows

- **Inbound (OpenClaw → ADE)**:
  - OpenClaw hook or skill calls ADE's single bridge entrypoint.
  - Payload includes the user message plus optional `targetHint` (`cto` or `agent:<worker-slug>`).
  - Bridge validates token and idempotency, resolves the ADE target through `agentChatService.ensureIdentitySession()`, injects turn-scoped bridge context, and forwards the message into the appropriate ADE identity session.
  - If the target worker slug is unknown or unavailable, the bridge falls back to CTO and records the fallback reason in bridge history.

- **Async reply path**:
  - Hook requests return `202` immediately.
  - When the ADE turn completes, the bridge delivers the reply back through the OpenClaw Gateway WebSocket operator session using `chat.send` when a `sessionKey` is known and `agent` when only `agentId` routing is available.

- **Sync fallback path**:
  - Skill-only or local `curl` integrations use `POST /openclaw/query`.
  - The bridge waits for the ADE reply and returns a plain JSON response body to the caller.

- **Outbound (ADE → OpenClaw)**:
  - CTO or the orchestrator emits a bridged notification such as `mission_complete`, `ci_broken`, or `blocked_run`.
  - The bridge resolves configured notification routes, reuses remembered session routing when possible, and sends over the Gateway WebSocket operator connection.
  - If the socket is unavailable, outbound messages are queued in the local outbox and replayed after reconnect.

##### Configuration

In `.ade/local.secret.yaml` (gitignored):
```yaml
openclaw:
  enabled: false
  gatewayUrl: ws://127.0.0.1:18789
  gatewayToken: ${OPENCLAW_GATEWAY_TOKEN}
  bridgePort: 18791
  hooksToken: <shared-secret>
  deviceToken: <persisted-after-first-handshake>
  allowedAgentIds: ["main"]
  defaultTarget: cto
  allowEmployeeTargets: true
  notificationRoutes:
    - notificationType: mission_complete
      agentId: main
      enabled: true
```

##### Scope Boundary

ADE implements the bridge service. OpenClaw-side configuration (hooks, skills, agent setup) is user-managed and documented in `docs/features/CTO.md`.

**Implementation status:** ✅ Complete (2026-03-12). W9 is implemented on top of W8's shipped external-MCP substrate.

**Tests:**
- HTTP endpoint: inbound message acceptance (202 acknowledged), async CTO forwarding, response delivery within timeout.
- Idempotency: duplicate inbound message returns cached response without re-invoking CTO.
- WebSocket operator: handshake (challenge-nonce, connect, hello-ok), reconnection with persisted `deviceToken`, graceful reconnection on connection drop.
- Outbound: operator WebSocket delivers replies and notifications, queues offline sends, and flushes outbox after reconnect.
- Context bridge: outbound context is filtered by `openclawContextPolicy`, inbound context stays turn-scoped bridge metadata instead of auto-persisted memory.
- Configuration: `.ade/local.secret.yaml` openclaw section parsing, enabled/disabled toggle, secret isolation, context policy applied.
- Error handling: OpenClaw gateway unavailable (graceful degradation, UI shows disconnected), CTO unavailable (queued retry with 3 attempts), malformed inbound requests (reject with 400), timeout on CTO processing (504 returned).
- Fallback mode: skill-only bridge works without WebSocket, inbound HTTP works, outbound notifications disabled.
