# W10: .ade/ Portable State
> Source: [OpenClaw — "file as truth" principle](https://velvetshark.com/articles/openclaw-memory-masterclass) — config files on disk are the source of truth, DB syncs from them. [Paperclip — log integrity](https://github.com/paperclipai/paperclip/blob/main/doc/SPEC.md) — SHA-256 chain for append-only logs. [Symphony — dynamic config reload](https://github.com/openai/symphony/blob/main/SPEC.md) — file-watch for runtime config changes.

Dependencies: **W1** (CTO state files), **W6½** (memory engine).

Consolidates the project-level `.ade/` directory structure, git-tracking policy, and configuration split (tracked vs. secret). This workstream materializes the storage layout that W1 (CTO state) and W6 (memory) populate. It also adds startup validation, integrity checking, and migration tooling.

##### Git-Tracking Policy

- **Tracked** (committable): `cto/`, `agents/`, `templates/`, `context/`, `memory/`, `history/`, `skills/`, `reflections/`, `local.yaml` — tiny config/state files for version tracking and team sharing.
- **Gitignored** (machine-specific): `local.secret.yaml`, `ade.db*`, `mcp.sock`, `embeddings.db`, `artifacts/`, `transcripts/`, `cache/`, `worktrees/`, `secrets/`.

##### Configuration Split

- `local.yaml` (tracked): Lane templates, phase profiles, feature flags, Linear sync config (project slug, dispatch rules, concurrency limits), automation rules, agent model preferences. No secrets.
- `local.secret.yaml` (gitignored): API keys, external MCP server configs, Linear API token, OpenClaw gateway token, local paths, OAuth tokens.

##### App State Sync Strategy

Git tracks code, config, and portable agent state. cr-sqlite (Phase 6) syncs app state (`ade.db`). Embeddings are local-only and regenerated on new machines. Memory JSON files are the "file as truth" source — DB syncs from files on startup if they diverge.

##### Startup Validation

On app startup, `adeProjectService.ts` validates the `.ade/` directory:

1. **Structure check**: Verify all expected directories exist. Create missing ones with correct permissions.
2. **Schema validation**: Validate `local.yaml` against a JSON Schema. Warn on unknown keys, error on invalid types. Validate all `identity.yaml` files against the `AgentIdentity` schema.
3. **Secret detection**: Scan tracked files for potential secrets (API keys, tokens). Warn if secrets found in tracked files — suggest moving to `local.secret.yaml`.
4. **Integrity check**: For append-only files (`missions.jsonl`, `sessions.jsonl`), verify each line is valid JSON. Truncate corrupted trailing entries with a warning.
5. **File-as-truth sync**: Compare on-disk identity files against DB `agents` table. If disk is newer (by mtime), update DB from disk. If DB is newer (no file), write file from DB. Log sync actions.
6. **Missing secret warning**: If features require `local.secret.yaml` entries (e.g., Linear sync enabled but no Linear token), show a non-blocking banner: "Linear sync requires API token — [Configure in Settings]".

##### Log Integrity

Adapted from [Paperclip's SHA-256 log chain](https://github.com/paperclipai/paperclip/blob/main/doc/SPEC.md): append-only JSONL files include a hash chain for tamper detection.

- **Hash chain**: Each JSONL entry includes a `prevHash` field containing `sha256(previous_entry_json)`. The first entry uses `prevHash: "genesis"`.
- **Verification**: On startup, `logIntegrityService.ts` samples 10% of entries (or all if < 100 entries) and verifies the hash chain. Broken chain → warning in UI + diagnostic log, but does not block startup.
- **Scope**: Applied to `missions.jsonl` and `sessions.jsonl`. Not applied to `reflections/` (orchestrator-managed, already has its own integrity via `ORCHESTRATOR_OVERHAUL.md` Phase 7).

##### Dynamic Config Reload

Watch `local.yaml` for changes (using `fs.watch`) and hot-reload affected services without app restart:

- Lane templates → reload `laneTemplateService`
- Phase profiles → reload `phaseEngine`
- Automation rules → reload `automationService`
- Agent model preferences → reload on next activation (no mid-run changes)
- Feature flags → reload affected services

Debounce: 2-second window. Only reload services whose config sections actually changed (diff-based).

##### Migration

Move any existing state from pre-Phase-4 storage into the `.ade/` structure:

- **Context packs** → project memory entries (W6 handles this)
- **Agent state from DB-only** → `.ade/agents/<name>/identity.yaml` + `core-memory.json`
- **CTO state from DB-only** → `.ade/cto/identity.yaml` + `core-memory.json`
- **One-time migration**: Runs on first startup after Phase 4 ships. Sets a `migration_v4_completed` flag in DB. Idempotent — safe to re-run.

##### Service Architecture

```
adeProjectService.ts     — Directory structure management, startup validation, secret detection
logIntegrityService.ts   — Hash chain verification for append-only JSONL files
configReloadService.ts   — File-watch hot-reload for local.yaml, diff-based service reload
migrationService.ts      — One-time state migration from pre-Phase-4 storage
```
