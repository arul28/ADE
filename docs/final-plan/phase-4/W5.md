# W5: Automations platform

W5 makes Automations a first-class product surface for creating, running, and auditing background workflows.

## Model contract

Automations follow one normalized model:

- Trigger categories: `time-based` or `action-based` only.
- Execution types: `agent-session`, `mission`, or `built-in task`.
- Budget policy: centralized in `Settings > Usage` (not per-rule budget policy).
- Output destinations:
  - `agent-session` output is written to Automations history.
  - `mission` output is written to Missions.
  - `built-in task` output is written to Automations history.

Removed concepts:

- Overnight queue workflows
- Morning briefing workflows
- Legacy executor modes that predate the current execution-type model

## Linear boundary

CTO owns Linear intake and dispatch policy. Automations can still invoke Linear as an action target, but Automations do not own issue-intake routing.

## Runtime shape

Automations still route through the orchestrator/runtime substrate for consistency, with execution metadata capturing trigger class and execution type.

```typescript
interface AutomationRule {
  id: string;
  name: string;
  trigger: {
    category: "time-based" | "action-based";
    config: Record<string, unknown>;
  };
  execution: {
    type: "agent-session" | "mission" | "built-in task";
    targetId?: string;
  };
  prompt?: string;
  toolPalette: string[];
  guardrails: {
    maxDurationMin?: number;
    verifyBeforePublish: boolean;
  };
}
```

## Surface model

- `/automations` is the authoring and operations surface.
- `Settings > Usage` is the single budget policy surface.
- Automations history is the audit surface for automation runs.
- Missions remains the audit/output surface for mission execution results.

## Implementation status

Complete:

- Trigger normalization to `time-based` and `action-based`.
- Execution routing to `agent-session`, `mission`, and `built-in task`.
- Run history and output routing split across Automations history and Missions.
- Runtime/policy alignment with centralized usage and budget policy in `Settings > Usage`.
