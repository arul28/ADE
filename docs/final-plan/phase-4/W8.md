# W8: External MCP Consumption
> Source: [MCP Specification](https://modelcontextprotocol.io/) — transport protocols, tool discovery, lifecycle management. [Paperclip adapters §4](https://github.com/paperclipai/paperclip/blob/main/doc/SPEC.md) — multi-adapter pattern for diverse tool backends. [Symphony §5 dynamic config](https://github.com/openai/symphony/blob/main/SPEC.md) — file-watch config reload without restart.

Dependencies: Independent infrastructure. Can begin any time after Phase 3.

ADE agents (mission workers and CTO) can consume external MCP servers, extending capabilities beyond what ADE provides natively. This is the gateway for third-party integrations — web browsing, Notion, Slack, GitHub, database tools, and any future MCP-compatible service.

##### Configuration

External MCP servers declared in `.ade/local.secret.yaml` (gitignored):
```yaml
externalMcp:
  - name: web-browser
    transport: stdio
    command: npx
    args: ["-y", "@anthropic/mcp-web-browser"]
    autoStart: true          # start on app launch
    healthCheckInterval: 30s # ping interval
  - name: notion
    transport: sse
    url: https://mcp.notion.so/sse
    headers:
      Authorization: "Bearer ${NOTION_TOKEN}"
    autoStart: false         # start on first use
  - name: linear
    transport: stdio
    command: npx
    args: ["-y", "@anthropic/mcp-linear"]
    autoStart: true
  - name: slack
    transport: stdio
    command: npx
    args: ["-y", "@anthropic/mcp-slack"]
    autoStart: false
    permissions:
      allowedTools: ["read_channel", "search"]  # restrict available tools
      blockedTools: ["send_message"]             # block dangerous tools
```

##### Dynamic Config Reload

Adapted from [Symphony's file-watch config reload](https://github.com/openai/symphony/blob/main/SPEC.md): watch `local.secret.yaml` for changes and hot-reload MCP server connections without app restart.

- **File watcher**: `fs.watch` on `.ade/local.secret.yaml`. On change, diff the `externalMcp` array against the running config.
- **Hot-reload behavior**:
  - New server added → connect and discover tools
  - Server removed → graceful disconnect (drain active tool calls, then shutdown)
  - Server config changed (URL, args, headers) → reconnect with new config
  - Permission changes → update tool registry immediately (no reconnect needed)
- **Debounce**: 2-second debounce on file changes to avoid rapid reconnection during editing.

##### Tool Discovery & Namespacing

- **Discovery**: On connection, call `tools/list` on each MCP server. Register discovered tools with namespace prefix: `ext.<server-name>.<tool-name>` (e.g., `ext.web-browser.navigate`, `ext.notion.search`).
- **Schema capture**: Store each tool's JSON Schema input definition for validation and UI rendering.
- **Re-discovery**: On reconnection, re-discover tools. If the tool set changed (new tools added, old tools removed), update the registry and notify connected agents.
- **Tool metadata enrichment**: Parse tool descriptions to categorize as "read" (safe) vs "write" (requires confirmation) for guardrail enforcement.

##### Permission Model

- **Per-server permissions**: Each MCP server config can specify `allowedTools` (whitelist) and `blockedTools` (blacklist). If both specified, `blockedTools` takes precedence.
- **Per-agent permissions**: Agent identity (W-UX) specifies which external MCP servers the agent can access:
  ```typescript
  // In AgentIdentity
  externalMcpAccess: {
    allowAll: boolean;              // CTO default: true, workers default: false
    allowedServers: string[];       // ["web-browser", "notion"]
    blockedServers: string[];       // ["slack"] — even if allowAll
  }
  ```
- **Mission-level MCP profile**: Mission launch flow includes an MCP tool selector. Selected tools are injected into worker context. Workers can only use tools approved for the mission.
- **Budget enforcement**: External tool invocations that incur API costs (e.g., web browsing tokens, Notion API calls) count against the agent's monthly budget. Cost estimation uses the tool's declared cost hint or a configurable default per server.

##### Lifecycle Management

- **Connection states**: `disconnected | connecting | connected | reconnecting | draining | failed`
- **Health checks**: Periodic `ping` to each connected server (configurable interval, default 30s). If 3 consecutive pings fail, mark as `failed` and attempt reconnection with exponential backoff (2s → 4s → 8s → max 60s).
- **Start modes**: `autoStart: true` starts on app launch. `autoStart: false` starts on first tool call (lazy initialization).
- **Graceful shutdown**: On app quit, send `shutdown` notification to all connected servers. Wait up to 5s for acknowledgment, then force-close.
- **Resource cleanup**: Stdio transport: kill child process. SSE transport: close HTTP connection.

##### Service Architecture

```
externalMcpRegistryService.ts    — Server config, connection registry, namespace management
externalMcpConnectionService.ts  — Transport lifecycle, health checks, reconnection
externalMcpPermissionService.ts  — Per-server + per-agent permission enforcement
externalMcpConfigWatcher.ts      — File-watch hot-reload for local.secret.yaml
```

##### Renderer

- **Settings > Integrations > External MCP**: Server list with connection status indicators (green/yellow/red dots). Per-server: tool list, permission config, health check stats, "Connect"/"Disconnect" buttons.
- **Settings > Integrations > "Add MCP Server"**: Guided flow — select transport (stdio/SSE), enter command/URL, test connection, configure permissions.
- **Mission launch > MCP tools**: Tool selector showing available external tools grouped by server. Pre-selected defaults from mission template.
- **CTO tab > Tools panel**: Available tools list (built-in + external) with usage stats per tool.
