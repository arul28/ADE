# W8: External MCP Consumption
> Sources: [MCP specification](https://modelcontextprotocol.io/specification/2025-06-18), [Anthropic Claude Code MCP docs](https://docs.anthropic.com/en/docs/claude-code/mcp), [OpenAI Codex MCP docs](https://platform.openai.com/docs/codex/mcp)

Dependencies: Independent infrastructure after Phase 3. This work is now implemented well enough to serve as the architectural baseline for **W9**.

## Status

**Implemented baseline.** ADE now owns external MCP configuration, discovery, permissioning, and tool exposure for ADE-managed sessions.

This is not a thin “let Claude/Codex talk to third-party MCP directly” feature. The shipped model is:

1. User configures external MCP in `.ade/local.secret.yaml`.
2. ADE connects to those external servers through `externalMcpService`.
3. ADE re-exposes approved external tools through ADE's own MCP server as namespaced tools like `ext.<server>.<tool>`.
4. ADE-launched Claude/Codex sessions still receive only ADE's MCP server at launch, but that ADE server now includes the filtered external tool surface.

That architecture keeps policy, budgeting, auditing, and mission scoping centralized inside ADE instead of pushing responsibility down into each provider runtime.

## Configuration

External servers live in `.ade/local.secret.yaml` under `externalMcp`:

```yaml
externalMcp:
  - name: github
    transport: stdio
    command: npx
    args: ["-y", "@modelcontextprotocol/server-github"]
    autoStart: true
    healthCheckInterval: 30s

  - name: notion
    transport: http
    url: https://mcp.notion.so/mcp
    headers:
      Authorization: "Bearer ${NOTION_TOKEN}"
    autoStart: false

  - name: legacy-browser
    transport: sse # accepted as a compatibility alias; resolved internally as http
    url: https://example.com/sse

  - name: slack
    transport: stdio
    command: npx
    args: ["-y", "@anthropic/mcp-slack"]
    permissions:
      allowedTools: ["read_channel", "search"]
      blockedTools: ["send_message"]
```

## Shipped Architecture

### Main services

- `apps/desktop/src/main/services/externalMcp/externalMcpService.ts`
  - Parses `externalMcp` config from `.ade/local.secret.yaml`
  - Supports `stdio` and streamable HTTP transport, with `sse` accepted as a compatibility alias
  - Handles connect/disconnect, lazy start, ping health checks, reconnect backoff, graceful shutdown, and tool discovery
  - Stores discovered tools as `ext.<server>.<tool>`

- `apps/mcp-server/src/mcpServer.ts`
  - Includes external tool manifests in `tools/list`
  - Dispatches `tools/call` for `ext.*` tools back through `externalMcpService`
  - Preserves structured MCP tool results instead of flattening them into ADE text wrappers

- `apps/mcp-server/src/bootstrap.ts`
  - Starts its own `externalMcpService` inside spawned ADE MCP server processes
  - Watches `.ade/local.secret.yaml` so active worker/CTO MCP sessions can hot-reload external config without restart

### Runtime injection model

- Claude and Codex workers are still launched with ADE MCP only.
- External MCP is available inside those sessions because ADE MCP now proxies approved external tools.
- Read-only Claude planning workers only auto-allow read-classified external tools.
- Worker owner ids are propagated into ADE-managed worker/subagent MCP sessions, so worker-specific external access policy and worker budget enforcement can apply.

### UI surfaces

- `Settings > Integrations > External MCP`
  - add/edit/remove/test/connect/disconnect
  - discovered tool list
  - recent usage events

- `CTO`
  - CTO-wide external MCP access policy
  - per-worker external MCP access policy

- `Mission launch / mission settings`
  - mission-level enable/disable
  - server selection
  - tool selection

## Permission Model

External MCP access is filtered in this order:

1. **Server config permissions**
   - `allowedTools`
   - `blockedTools`
   - block wins over allow

2. **Identity policy**
   - CTO identity: `externalMcpAccess`
   - Worker identity: `externalMcpAccess`

3. **Mission selection**
   - `permissionConfig.externalMcp`
   - can disable all external MCP for a mission or narrow to selected servers/tools

4. **Execution mode**
   - read-only planning workers only surface read-safe external tools into Claude's launch allowlist

## Dynamic Reload + Lifecycle

- Desktop app config reload path reuses `configReloadService.ts`
- Headless ADE MCP server sessions also watch `.ade/local.secret.yaml`
- External MCP reload is debounced inside `externalMcpService`
- Permission-only edits re-filter the tool map without forcing reconnect
- Signature-changing edits reconnect the external server
- Removal drains and disconnects the runtime

Connection states:

- `disconnected`
- `connecting`
- `connected`
- `reconnecting`
- `draining`
- `failed`

## Budgeting + Audit

- External MCP calls are checked against worker and mission budget gates before execution
- Worker cost events and recent external MCP usage events are recorded by ADE
- Tool safety classification and namespaced tool identity are preserved in usage metadata

## Current Baseline for W9

W9 should assume the following are now true:

- ADE is the single source of truth for external MCP config.
- External tools appear to ADE-managed runtimes as ADE MCP tools named `ext.<server>.<tool>`.
- W9 should build on ADE's permissioned external MCP substrate, not on provider-global MCP config files.
- If W9 needs OpenClaw-originated agent/tool access, it should either:
  - use ADE's MCP surface directly, or
  - call into ADE services that already resolve external MCP policy/config.

## Remaining Follow-Through

W8 is ready enough for W9, but a few non-blocking follow-through items remain outside the core substrate:

- renderer live-refresh for cross-panel external MCP state changes is still basic
- mission budget burn-down is gated pre-call, but external usage is not yet a dedicated persisted mission-cost ledger
- ADE MCP still behaves as a static outward tool list from the client's perspective; internal external-tool re-discovery is implemented, but outward `tools/list_changed` notifications are not yet a first-class flow
